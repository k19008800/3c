## 一、产品定位

3cloud（简称 3C）是一个 **AI Token 聚合中转平台**，上游对接多家 AI 厂商（DeepSeek、豆包、天翼云），下游面向个人开发者和企业客户，提供统一的 API 调用入口。平台自动（或用户手动指定）选择同一模型下价格最低的厂商提供服务，对下游用户完全隐藏上游细节。

### 核心价值

- **聚合比价** — 同一模型多家厂商竞品，自动走最低价
- **零门槛接入** — 用户改个 Base URL + API Key 即用，兼容 OpenAI SDK
- **代理商生态** — 内置代理商 / 分佣 / 微信提现体系
- **全链路可控** — 用量、计费、日志、告警，用户和管理员双视角透明

### 与 3cloud 的关系

完全独立，从零新开发。不复用 3cloud 代码、数据库、用户体系。

### 域名规划

| 域名 | 用途 |
|---|---|
| `unmisa.com` | Web 控制台（用户端 + 管理后台） |
| `api.unmisa.com` | Token 代理 API（兼容 OpenAI 格式） |

## 四、功能模块

### 4.1 用户体系

#### 注册流程

1. 用户填写邮箱 + 密码 + 确认密码（密码 ≥ 6 位）
2. 校验邮箱唯一性 → 生成用户记录（status: `pending`）
3. 发送验证邮件（24h 有效期，60 秒内仅可重发一次）
4. 用户点击验证链接 → 激活账号

#### 用户状态与生命周期 ✅ 已决策

```
注册 → PENDING（待验证邮箱）
  │
  ├── 验证邮箱（24h 内点击链接）
  │       │
  │       ▼
  │   ACTIVE（已验证邮箱，可正常使用）
  │       │
  │       ├── 可跳过实名，以普通用户身份正常使用
  │       │   （赠送体验额度，不受限）
  │       │
  │       ├── DISABLED（管理员封禁）
  │       │   ├─ 可登录，可见余额
  │       │   ├─ 不可发起新请求，API Key 鉴权返回 401
  │       │   ├─ 已发起的流式请求允许走完
  │       │   ├─ 支持 N 天后自动解封或管理员手动解封
  │       │   └─ 操作记入 audit_logs
  │       │
  │       └── DELETED（用户注销）
  │           ├─ 软删除，数据保留
  │           ├─ 不可登录，不可重新注册
  │           └─ 余额 > 0 需走人工退款
  │
  └── 验证链接过期 → 保留 PENDING，可重新发送
```

**状态枚举（pgEnum）：** `pending` / `active` / `disabled` / `deleted`

**关键规则：**

| 状态 | 可登录 | 可发起 API 请求 | 余额可见 | 实名审核 | 充值 |
|---|---|---|---|---|---|
| PENDING | ❌ | ❌ | ❌ | ❌ | ❌ |
| ACTIVE | ✅ | ✅ | ✅ | 可选提交通道 | ✅ |
| DISABLED | ✅（只读） | ❌ | ✅（可见） | ❌ | ❌ |
| DELETED | ❌ | ❌ | — | — | — |

**解耦设计：** 邮箱验证和实名认证互不绑定。用户验证邮箱即进入 ACTIVE，实名认证是独立可选步骤，不卡功能。

#### 登录

- 邮箱 + 密码 → JWT 双 Token
- Access Token：2h 有效期
- Refresh Token：7d 有效期
- 支持多设备同时登录

#### 实名认证

| 类型 | 材料要求 |
|---|---|
| **个人** | 真实姓名 + 身份证号 + 身份证正反面图片（≤5MB） |
| **企业** | 公司全称 + 统一信用代码 + 营业执照（≤10MB）+ 开户行 / 账号 / 地址 + 开票资料 |

- 提交后 status: `pending_review`
- 管理员审核 → 通过 / 驳回（附驳回原因）
- 驳回后可修改重新提交

**✅ 已决策（一次性认证，不支持自主更新）：** 实名通过后锁定状态，不可在线修改。如需更新材料（换身份证、营业执照到期等），联系客服 → 管理员后台手动重置实名状态至待审核。用户控制台展示

#### 个人中心

- 查看 / 修改个人资料
- 修改密码
- 余额查看

#### 账户注销

- 软删除，数据保留
- 余额 > 0 需先人工退款
- 同一邮箱不可重新注册

#### 密码重置

- 支持重发重置邮件
- 60 秒频率限制

- 格式：`sk-` + 随机字符串

### 4.5 模型 & 厂商管理（后台 CRUD）

#### 厂商管理

- 内置：**DeepSeek / 豆包 / 天翼云**
- 支持管理后台新增厂商

#### 模型管理

- 类型枚举：`chat` / `embedding` / `image` / `audio`

#### 厂商-模型映射

每个模型可关联多个厂商，每个厂商关联配置：

| 配置项 | 说明 |
|---|---|
| API 端点 | 厂商端 URL |
| API Key | 厂商端 Key（可配多个，自动分摊，**明文存储** ✅ BOSS 确认） |
| 成本价 | 平台向上游支付的价格（每 token） |
| 售价 | 平台向下游收取的价格（每 token） |
| 权重 | 加权路由分配权重 |

**示例：**

```
模型: deepseek-v4-pro
├── DeepSeek  成本 ¥0.60  → 售价 ¥0.80  (模型名: deepseek-chat)     (权重 50)
├── 天翼云    成本 ¥0.55  → 售价 ¥0.73  (模型名: deepseek-v4-pro)  (权重 100)  ← 自动选此厂商！
```

> **模型名映射说明：** 用户传的统一模型名（如 `deepseek-v4-pro`）→ 硬映射到上游厂商的真实模型名（如 DeepSeek 官方为 `deepseek-chat`），在后台厂商-模型映射表中配置。

### 4.6 智能路由

#### 策略

| 策略 | 说明 |
|---|---|
| **自动最低价（默认）** | 每次请求自动选最低价可用厂商 |
| **手动指定** | 可将用户绑定到指定厂商 |
| **加权动态分配** | 按 weight 加权随机分配（低价高权重，高价也有流量） |
| **故障切换** | 厂商不可用时自动切换到次低价厂商 |
| **多 Key 分摊** | 同一厂商的多 Key 自动分摊消耗 |

#### 健康检查

| 类型 | 机制 |
|---|---|
| **被动检查（主要）** | 近 50 次请求中，成功率 < 70% → 降级（权重降至 50%）；< 30% → 宕机（权重归零）。计数器存在 **Redis Sorted Set** 中，服务重启不丢失。✅ 已决策 |
| **主动检查（辅助）** | 每 5 分钟对 down 厂商发轻量请求，连续成功 3 次 → 恢复 |

#### 上游异常处理

| 异常 | 处理 |
|---|---|
| 超时 > 10s | 告警 + 自动切换 |
| HTTP 429 / 5xx | 切换到备用厂商 |

#### 路由逻辑

```
请求 → API Key 鉴权 → 余额检查 → 模型解析
→ 厂商路由（加权动态分配，跳过 down 厂商，兜底 503）
→ 代理转发（替换 model 名 + API Key）
→ 流式 / 非流式响应（余额耗尽允许走完）
→ 计费（usage × 价格 × 折扣）
→ 分佣（如归属代理商）
```

### 4.7 计费系统

#### 定价公式

```
最终售价 = 模型公开售价 × 定价倍率 × 折扣系数
```

- 模型售价：管理后台按 `input_token` / `output_token` 分别设定
- 定价倍率：默认 **1.33**，后台可配置
- 折扣系数：个人 / 企业独立折扣（企业折扣覆盖个人折扣）

#### 充值

| 渠道 | 说明 |
|---|---|
| 微信支付 | PC 扫码 / 手机 JSAPI |
| 支付宝 | PC 扫码 / 手机 JSAPI |
| 对公转账 | 上传凭证 → 管理员后台确认入账（线下入账审核） |

- 最低 1 元起充，无赠送
- 30 分钟未支付 → 自动取消，幂等处理
- **注册用户统一折扣：** 后台系统配置中维护默认折扣率
- **✅ 已决策（分档风控）：**
  - 个人用户单次充值上限 ¥5,000
  - 企业用户单次充值上限 ¥50,000
  - 单日累计超 ¥20,000（个人）/ ¥200,000（企业）触发风控标记，需人工审核
  - 风控阈值在 `system_configs` 中可配置

#### 扣费

- **实时扣费**，流式请求结束后一次性按总 usage 扣费（边流边累加，流毕统一结算）
- 流式请求中途余额用尽 → 允许走完（余额可为负值）
- 流式请求中途断连 → 已消费部分**不计费**
- 余额为负状态下**不可发起新请求**，必须先充值回补到 0 以上
- 下次充值优先回补负余额
- **价格精度：** 0.000001（六位小数）
- **退款：** 不设自动退款，人工处理（仅退未使用余额）

#### 双阈值告警

| 阈值 | 行为 |
|---|---|
| 系统默认阈值 | 平台统一设定（系统配置中可调） |
| 用户自定义阈值 | 用户自行设定 |
| 触发条件 | 余额低于**任意一个阈值**即告警 |

**✅ 已决策（按月分区方案）：** `call_logs` 表按 `created_at` 做 PG 原生表分区 `PARTITION BY RANGE`，每个月一个分区。90 天后直接 DROP 旧分区（比 DELETE 快很多），不建独立聚合表。分区裁剪确保查询性能。

## 八、核心业务流程

### 8.1 API 调用链路

```
请求 → API Key 鉴权 → 余额检查 → 模型解析
→ 厂商路由（加权动态分配，跳过 down 厂商，兜底 503）
→ 代理转发（替换 model 名 + API Key）
→ 流式 / 非流式响应（余额耗尽允许走完）
→ 计费（usage × 价格 × 折扣）
→ 分佣（如归属代理商）
```

### 8.2 充值流程

```
选择金额 / 支付方式 → 生成订单
  ├── 在线支付 → 支付宝/微信回调 → 更新余额
  └── 对公转账 → 上传凭证 → 管理员确认入账
30 分钟未支付 → 自动取消（幂等）
```

### 8.3 代理商提现流程

```
代理商提交申请（微信提现）
        │
   管理员审核
        │
   ┌────┴────┐
   ▼         ▼
 通过        驳回
   │           │
微信企业付款   通知原因
   │
更新状态
```

### 8.4 加权路由策略

```
输入：模型名, 用户信息
步骤：
  1. 查询该模型下所有可用厂商-模型关联
  2. 筛选健康（被动+主动检查）且余额充足的 Key
  3. 选择路由策略：
     a. 自动最低价 → 选最低价厂商
     b. 加权动态 → 按权重加权随机选
     c. 手动指定 → 走指定厂商
  4. 若选中的 Key 调用失败：
     a. 被动健康检测 → 降级权重
     b. 自动切换到下一个备选
  5. 全部不可用 → 返回 503
```

## 十四、需求增补（项目经理审计 2026-06-27）

> 以下为 PRD 定稿后由项目经理逐章审计发现的补充需求，共 **22 项**。

### 🔴 必须解决（6 项）

#### 1. 余额字段精度 — 使用 DECIMAL 而非 float

- 文档要求精度 0.000001（六位小数）
- **必须使用 `NUMERIC(18,6)` 或 `DECIMAL(18,6)` 类型** 存储余额相关字段
- 涉及：`users.balance`、`call_logs.cost`、`balance_logs.amount`、`commission_logs.amount` 等
- **影响：** float 存在浮点误差，累计后可能导致账务不平

#### 2. 调用日志表索引方案

- `call_logs` 表 90 天全量明细 + 91~365 天聚合，预计百万级至千万级数据量
- **必须预定义索引策略：**
  - `(user_id, created_at DESC)` — 用户查自己的日志
  - `(api_key_id, created_at DESC)` — 按 Key 级追溯
  - `(vendor, created_at DESC)` — 厂商维度统计
  - `(status, created_at DESC)` — 失败请求排查

#### 3. SSE 后端超时硬限制

- 文档定义前端 10s 超时切厂商，但后端缺少硬超时兜底
- **后端需实现：**
  - Fastify 请求级别的超时中间件（如 `@fastify/timeout`）
  - 向上游厂商发起请求的超时设置（`AbortController.timeout`）
  - 超时后触发被动健康检测降级

#### 4. 管理员操作审计日志

- 所有后台管理操作必须记录审计日志
- 审计范围：
  - 用户创建 / 封禁 / 重置密码
  - 余额变更（人工充值 / 扣费回退）
  - 角色变更（user→agent、admin 降级等）
  - 系统配置修改（限流值 / 定价倍率 / 邮件 SMTP）
  - 代理商提现审核（通过 / 驳回）
  - 实名审核（通过 / 驳回）
- **存储：** 独立 `audit_logs` 表，记录操作人、操作类型、目标对象、变更前后值、IP、时间
- **保留：** 永久保留

#### 5. 厂商 API Key 加密存储（建议替代方案）

- BOSS 确认明文存储。但安全风险不可忽视，建议折中方案：
- **AES-256-GCM 应用层加密**，密钥存环境变量 `VENDOR_KEY_ENCRYPTION_KEY`
- 运行时解密到内存，不落日志
- 管理员在后台维护明文 → 后端加密后存 DB → 读取时解密
- **如坚持明文：** 需要在 DB 层面限制 `vendor_models` 表的访问权限，且备份文件需加密

#### 6. 数据库 Migration 回滚方案

- Drizzle ORM 生成 migration 文件 → 上生产
- **必须定义回滚流程：**
  - `drizzle-kit up` 灰度：先在 staging 执行，再上生产
  - `drizzle-kit down` 回滚：每次 migration 必须有对应的 down SQL
  - 上线前执行 `pg_dump` 全量备份，回滚失败时直接恢复到备份点
