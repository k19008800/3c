# 交互性能瓶颈分析 - 关键发现摘要

## 🔍 核心发现

### 1. 高频交互路径
- **Token代理路径**: 每天数万次调用，链路过长（9个步骤）
- **用户管理路径**: 管理后台高频使用，缺少缓存
- **导出操作**: 全量数据导出，内存和性能压力大

### 2. 关键性能数据
```
前端API调用统计:
- 总计33个API调用点
- fetch()直接调用32处，axios封装1处
- 11个页面有API调用，平均每页3个调用
- 用户管理页面13个调用点（最多）

后端路由统计:
- 150个路由文件，571+个端点
- 183个for...of循环存在N+1查询风险
- 169处JSON.parse()存在同步阻塞风险

定时任务:
- 7个核心定时任务，最高频率每小时
- 对账任务预估30分钟，影响最大
```

### 3. 优先级排序

**P0（立即修复）**:
1. 代理商概览N+1查询（200次→1次批量查询）
2. `call_logs`大表COUNT性能优化
3. Redis KEYS命令替换为SCAN
4. 导出接口流式改造

**P1（短期优化）**:
1. 管理列表添加Redis缓存（5分钟TTL）
2. 前端请求合并/去重
3. 定时任务异步队列化
4. JSON解析分批处理

**P2（架构改进）**:
1. 服务拆分（代理/管理/定时任务分离）
2. 数据库读写分离
3. 完整监控体系
4. 性能测试常态化

## 📊 性能影响预估

| 优化项 | 当前状态 | 优化后 | 预期提升 |
|--------|----------|--------|----------|
| N+1查询修复 | 200次查询/页面 | 1次批量查询 | 99.5% |
| 大表COUNT | 全表扫描 | 分区计数缓存 | 90%+ |
| 管理列表缓存 | 每次DB查询 | 80%缓存命中 | 80% |
| 导出接口 | 全量内存加载 | 流式分批 | 内存减80% |

## 🚀 立即行动项

### 代码层面
```typescript
// 1. N+1查询修复示例
// 当前: 循环内逐个查询
for (const agent of agents) {
  const stats = await db.query.agentStats(agent.id);
}

// 优化: 批量预加载
const agentIds = agents.map(a => a.id);
const allStats = await db.query.agentStats
  .where(agentId in agentIds)
  .execute();

// 2. Redis Pipeline优化
// 当前: 逐个请求
for (const userId of userIds) {
  await redis.exists(`risk:ban:user:${userId}`);
}

// 优化: Pipeline批量
const pipeline = redis.pipeline();
userIds.forEach(id => pipeline.exists(`risk:ban:user:${id}`));
const results = await pipeline.exec();
```

3. **Node.js配置**: 增加事件循环监控，调整GC参数
